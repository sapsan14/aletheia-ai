| Область | Статус |

## Позиционирование

> **Aletheia AI предоставляет криптографически верифицируемые утверждения ИИ с доверенными временными метками — для аудита, юридического и регуляторного использования.**

Так мы выходим за рамки «безопасности чат-бота» к **AI attestation** и **инфраструктуре доверия**.

## Расширенное видение

Помимо аттестации ответов, инфраструктура доверия может распространяться на:

| Область | Что аттестуем | Зачем |
|---------|---------------|-------|
| **Модели** | Веса модели, архитектура, целостность чекпоинта | Воспроизводимость, supply chain; traceability EU AI Act для high-risk систем |
| **Агенты** | Идентичность агента, возможности, использование инструментов | Мульти-агентные системы; кто что сделал |
| **Датасеты и данные** | Провенанс обучающих данных, lineage, consent | Compliance, аудит bias, авторские права |
| **AI-to-AI через MCP** | Аттестация поверх Model Context Protocol | Один агент сертифицирует другого; interoperable AI trust layer |

**Подпись весов нейросети** — криптографическое связывание checkpoint модели с конкретной версией и провенансом. Верификатор может подтвердить точную модель, породившую результат.

Это стратегические перспективы: от «подписанных ответов» к **full-stack AI trust** — модели, данные, агенты, ответы. Ближний roadmap остаётся на аттестации ответов; это горизонт.

### Смежные слои доверия

Доверие к **действию** (кто что сделал, когда) дополняется другими слоями. **Proof of Human** (например [World ID](https://worldcoin.org) / Orb и аналоги) отвечает на вопрос «кто человек» — без раскрытия персональных данных. Aletheia отвечает на «что именно сделано, кем и когда». Это ортогональные оси: один слой — идентичность человечности, другой — криптографическое доказательство результата. Proof of Human может использоваться как опциональный вход при выпуске сертификатов агентов (например, «агент под контролем верифицированного человека»). Подробнее: [идея: World ID и Aletheia](ideas/WORLD_ID_AND_ALETHEIA.md).

## Стратегическая дорожная карта

Единый приоритизированный план (фазы по влиянию и зависимостям):

| Фаза | Фокус | Цель |
|------|-------|------|
| **1** | Evidence Package + офлайн-верификация | Trust infrastructure; проверка без нашего сервера |
| **2** | Killer demo (legal/compliance) | Product-market fit; история для инвесторов; выбор ниши |
| **3** | AI Claim (структурированная аттестация) | Audit-grade утверждения с контекстом и confidence |
| **4** | Модель ключей и доверия | PKI for AI; key_id, registry, ротация |
| **5** | Анкорирование доверенного времени | Multi-TSA, публичные якоря (Bitcoin, Ethereum, Roughtime) |
| **6** | Production: облако + путь к HSM | Масштабируемость, SLA, enterprise readiness |

**Будущие перспективы:**

- **AI vs AI верификация** — один ИИ утверждает, другой проверяет; multi-agent trust (связь с MCP и расширенным видением)
- **EU-style roadmap** — PoC → Pilot → выравнивание с регуляторикой

## Фокусные области (подробно)

### 1. Evidence Package и офлайн-верификация

**Сейчас:** Проверка только на нашем backend.

**Цель:** Проверка **без** нашей системы.

| Результат | Назначение |
|-----------|------------|
| **Evidence Package (.aep)** | response.txt, canonical.bin, hash.sha256, signature.sig, timestamp.tsr, metadata.json, public_key.pem |
| **CLI** | `aletheia verify response.json` — локальный запуск |
| **Чистый JS-верификатор** | Браузер или Node — без вызова backend |

Аудиторы, суды и компании могут проверить хеш, подпись и timestamp офлайн. Это момент, когда проект становится **trust infrastructure**.

---

### 2. Killer demo и выбор ниши

Важно выбрать **одну** приоритетную нишу. Рекомендуется: **legal / compliance AI** (EU AI Act, контракты, аудит).

| Ниша | Почему |
|------|--------|
| **Legal / compliance AI** | Контракты, оговорки, регуляторные проверки |
| **Medical AI opinions** | Второе мнение, документация |
| **AI-generated contracts** | Подписание и аттестация AI-договоров |
| **Scientific AI results** | Воспроизводимость, цитирование |
| **AI journalism / fact claims** | Верифицируемые источники и утверждения |

Важно: не чат, а **утверждения с последствиями**.

---

### 3. AI Claim (доказуемое утверждение)

**Сейчас:** «ИИ ответил вот так, и это подписано + timestamped».

**Цель:** «ИИ сделал **утверждение X** при **условиях Y**, и это **доказуемо**».

Подписывать структурированное **утверждение** и контекст вместо сырого текста:

```json
{
  "claim": "Эта оговорка контракта соответствует GDPR",
  "confidence": 0.82,
  "reasoning_hash": "...",
  "model": "gpt-4.1",
  "policy_version": "gdpr-2024-05"
}
```

| Аспект | Польза |
|--------|--------|
| **Подписанное содержимое** | Утверждение + контекст (модель, версия политики) |
| **Confidence** | Явная неопределённость для legal/compliance |
| **Версия политики** | Аудит: какие правила действовали |

---

### 4. Модель ключей и доверия

**Сейчас:** По сути один ключ для подписи.

**Цель:** Явная **модель ключей и доверия** — PKI for AI.

| Элемент | Описание |
|---------|----------|
| **key_id** | В каждой записи; связь с публичным ключом |
| **Публичный key registry** | JSON или JWKS |
| **Метаданные** | purpose, algorithm, valid_from / valid_to |
| **Дальше** | Ротация ключей, key provenance, разные ключи под legal, medical, financial AI |

**PKI / CA опции** — ключи подписи могут выпускаться и управляться established PKI:

| Решение | Назначение |
|---------|------------|
| **Smallstep (step-ca)** | Лёгкий CA; ACME, OIDC; short-lived certs; DevOps-friendly |
| **EJBCA** | Enterprise PKI; CA/RA/VA; Common Criteria; регулируемые отрасли |
| **HashiCorp Vault** | Transit engine: подпись хешей через API (ключи не покидают Vault); PKI engine для выпуска cert; secrets для API keys, DB, TSA creds |

**HashiCorp Vault на практике** — Backend вызывает Vault Transit для подписи хеша ответа вместо локального ключевого файла. Ключи остаются в Vault; подпись — это API-вызов. Vault может выпускать X.509 cert (PKI engine) и хранить секреты (OpenAI key, TSA config). Распространён в enterprise; дополняет или заменяет HSM для software-based защиты ключей.

→ Сертификаты подписи Aletheia — Smallstep/EJBCA; или подпись через Vault Transit / HSM; полная цепочка доверия, revocation, lifecycle.

См.: [Модель доверия и eIDAS](TRUST_MODEL.md).

### 5. Доверенное время, не просто метка

**Сейчас:** TSA = техническая функция «время подписания».

**Цель:** **Доказательство момента в истории** — доверенное время.

| Механизм | Назначение |
|----------|------------|
| **Multi-TSA** | 2–3 независимых TSA для избыточности и доверия |
| **Публичные якоря** | Bitcoin block hash, Ethereum calldata, RFC 9162 Roughtime (опционально) |

Итог: «Этот ответ ИИ **существовал до** события X» — юридически и научно сильно.

См.: [Временные метки](TIMESTAMPING.md).

### 6. AI vs AI верификация (будущее)

Один ИИ отвечает; другой ИИ **проверяет** утверждение. Оба ответа подписаны и с временной меткой.

| Роль | Пример |
|------|--------|
| **AI-1** | «Этот код безопасен». |
| **AI-2** | «Подтверждаю: утверждение корректно». |

→ **Multi-agent trust** — ближе к научной верификации.

**Расширение:** Идея PKI для автономных AI-агентов (OpenClaw, агенты в экосистеме MCP, верификация и сертификация с нашей PKI) описана в отдельном документе: [Идея: PKI для AI — использование и тестирование](ideas/PKI_FOR_AI_AGENTS.md).

## Production: облако и HSM

**Облако:** Стек (Docker, Spring Boot, Next.js) готов к облаку. Рекомендуется: AWS (ECS/EKS, RDS), GCP (Cloud Run, Cloud SQL) или Azure (Container Apps, PostgreSQL).

**HSM:** Сейчас — файловый ключ `ai.key`. Production — подпись через HSM (AWS KMS, CloudHSM, Azure Key Vault HSM, Google Cloud HSM). Абстракция `SignatureService` — один impl для файла, другой для HSM/KMS. «Enterprise: переход с файла на HSM одной сменой конфига».

## Pitch для инвесторов

> **Aletheia AI — Trust Infrastructure for AI**
>
> Превращаем выводы ИИ в криптографически верифицируемые доказательства: подписаны, RFC 3161 timestamped, верифицируемы офлайн.
>
> **Проблема:** Невозможно доказать, что сказал ИИ, когда и в каком контексте.
>
> **Решение:** Подписанные утверждения ИИ с доверенными временными метками; верификация без нашего сервера.
>
> **Рынок:** Legal tech, compliance, fintech, регуляторы (EU AI Act).
>
> **Статус:** PoC готов; RFC 3161; архитектура совместима с eIDAS; путь к HSM и qualified TSA.
>
> **Дальше:** Evidence Package, офлайн-верификация, пилот в legal/compliance.

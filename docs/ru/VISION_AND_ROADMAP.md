## Текущее состояние: что уже есть

| Область | Статус |
|--------|--------|
| **PoC** | ✅ Готов |
| **Демо** | ✅ Работает сквозной сценарий |
| **Крипто-пайплайн** | ✅ Canonicalize → hash → sign → timestamp (RFC 3161) |
| **RFC 3161 TSA** | ✅ Реализован (редкость для AI-проектов) |
| **Цепочка доверия** | ✅ Подписанный ответ + TSA-токен; проверка на нашем backend |
| **Деплой** | ✅ Docker, docker-compose, Ansible |

Закрыта самая сложная часть: **криптографически строгая цепочка доверия для ИИ**. Многие стартапы ломаются именно здесь. Следующий уровень — превратить «подписанный ответ ИИ» в **оформляемое, audit-grade AI attestation** и **проверку без нашего сервера**.

## Позиционирование для внешнего мира

Уже можно формулировать:

> **Aletheia AI предоставляет криптографически верифицируемые утверждения ИИ с доверенными временными метками — для аудита, юридического и регуляторного использования.**

Так мы выходим за рамки «безопасности чат-бота» к **AI attestation** и **инфраструктуре доверия**.

## Большой шаг 1: AI Claim (доказуемое утверждение)

**Сейчас:** «ИИ ответил вот так, и это подписано + timestamped».

**Дальше:** «ИИ сделал **утверждение X** при **условиях Y**, и это **доказуемо**».

### Claim object (формализованный ответ)

Подписывать не просто текст, а структурированное **утверждение** и контекст:

```json
{
  "claim": "Эта оговорка контракта соответствует GDPR",
  "confidence": 0.82,
  "reasoning_hash": "...",
  "model": "gpt-4.1",
  "policy_version": "gdpr-2024-05"
}
```

| Аспект | Польза |
|--------|--------|
| **Подписанное содержимое** | Утверждение + контекст (модель, версия политики) |
| **Confidence** | Явная неопределённость для legal/compliance |
| **Версия политики** | Аудит: какие правила действовали |
| **Итог** | Уровень legal / compliance / audit |

→ Выход за пределы «chatbot security» в **AI attestation**.

## Большой шаг 2: Верификация вне нашей системы

**Сейчас:** Проверка = наш backend.

**Дальше:** Проверка **без** нашей системы.

### Офлайн-верификация

| Результат | Назначение |
|-----------|------------|
| **CLI** | `aletheia verify response.json` — локальный запуск |
| **Чистый JS-верификатор** | Браузер или Node — без вызова backend |
| **Экспорт-пакет** | `response.json`, `signature.pem`, `tsa.tsr`, `public_key.pem` |

Любой аудитор, суд или компания может проверить **хеш**, **подпись** и **timestamp** без нашего сервера. Это момент, когда проект становится **trust infrastructure**.

---

## Большой шаг 3: Модель ключей и доверия

**Сейчас:** По сути один ключ для подписи.

**Дальше:** Явная **модель ключей и доверия**.

### Минимум

| Элемент | Описание |
|---------|----------|
| **key_id** | В каждой записи; связь с публичным ключом |
| **Публичный key registry** | JSON или JWKS |
| **Метаданные** | `purpose`, `algorithm`, `valid_from` / `valid_to` |

### Дальше

- **Ротация ключей** без поломки старых доказательств
- **Key provenance** — кто и зачем выпустил ключ
- **Разные ключи** под legal, medical, financial AI

→ Это уже **PKI for AI**, а не просто подпись.

См. также: [Модель доверия и eIDAS](TRUST_MODEL.md).

## Большой шаг 4: Доверенное время, не просто метка

**Сейчас:** TSA = техническая функция «время подписания».

**Дальше:** **Доказательство момента в истории** — доверенное время.

### Идеи

| Механизм | Назначение |
|----------|------------|
| **Multi-TSA** | 2–3 независимых TSA для избыточности и доверия |
| **Якорь в публичные источники** | Bitcoin block hash, Ethereum calldata или RFC 9162 Roughtime (опционально) |

Итог: «Этот ответ ИИ **существовал до** события X» — юридически и научно сильно.

См. также: [Временные метки](TIMESTAMPING.md).

## Большой шаг 5: AI Evidence Package

Собирать всё в один **пакет доказательств**:

```
Aletheia Evidence Package (.aep)
├── response.txt
├── canonical.bin
├── hash.sha256
├── signature.sig
├── timestamp.tsr
├── metadata.json
└── public_key.pem
```

### Использование

- Суды
- Аудит
- Регуляторы
- Корпоративные расследования
- Научные публикации

→ Создаётся **новый формат доказательств для ИИ**.

## Большой шаг 6: AI vs AI верификация

**Идея:** Один ИИ отвечает; другой ИИ **проверяет** утверждение. Оба ответа подписаны и с временной меткой.

| Роль | Пример |
|------|--------|
| **AI-1** | «Этот код безопасен». |
| **AI-2** | «Подтверждаю: утверждение корректно». |

→ **Multi-agent trust** — ближе к научной верификации.

## Большой шаг 7: Выбор killer-ниши

Готовность действовать как **архитектор**, а не только технарь. Важно выбрать **одну** приоритетную нишу.

### Сильные кандидаты

| Ниша | Почему |
|------|--------|
| **Legal / compliance AI** | Контракты, оговорки, регуляторные проверки |
| **Medical AI opinions** | Второе мнение, документация |
| **AI-generated contracts** | Подписание и аттестация AI-договоров |
| **Scientific AI results** | Воспроизводимость, цитирование |
| **AI journalism / fact claims** | Верифицируемые источники и утверждения |

Важно: не чат, а **утверждения с последствиями**.

## Облако и HSM

### Облачный деплой

Стек (Docker, Spring Boot, Next.js) готов к облаку. Рекомендуемый путь:

| Облако | Сервисы |
|--------|---------|
| **AWS** | ECS/EKS, RDS PostgreSQL, опционально CloudHSM |
| **GCP** | Cloud Run, Cloud SQL |
| **Azure** | Container Apps, PostgreSQL |

→ Один production-деплой в облаке для масштабируемости, SLA и доверия инвесторов.

### HSM (Hardware Security Module)

**Сейчас:** Файловый ключ подписи (`ai.key`).

**Production / enterprise:** Подпись через HSM для юридической силы и compliance.

| Вариант | Назначение |
|---------|------------|
| **AWS KMS** | Managed; подпись через API |
| **AWS CloudHSM** | Выделенный HSM; полный контроль |
| **Azure Key Vault HSM** | Managed HSM backing |
| **Google Cloud HSM** | Cloud HSM module |

**Путь:** Абстракция `SignatureService` — один impl для файла, другой для HSM/KMS. Документация: «Enterprise: переход с файла на HSM одной сменой конфига.»

→ Архитектура поддерживает HSM; enterprise-клиенты могут подключить свой.
